소프트웨어 (Software) - 심화

머신비전 개발자를 위한 소프트웨어 아키텍처 · 개발 심화 학습

1. 개요

  • 머신 비전 시스템에서 이미징 소프트웨어는 하드웨어와 응용 프로그램 사이를 연결하는 핵심 역할을 담당
  • 이미징 소프트웨어의 주요 역할
    • 카메라 하드웨어와의 인터페이스
    카메라 기능 제어 (노출, 게인, 트리거 등)
    이미지 수신, 처리 및 분석
    이미지 결과를 이용한 의사결정 수행
    다른 연결된 장치 또는 기계와 통신
개발자 노트 — 이 페이지의 모든 소프트웨어 계층은 결국 인터페이스 페이지에서 다룬 GenICam/GenTL 표준 위에서 동작합니다. 카메라 뷰어, 포괄적 소프트웨어, SDK 모두 내부적으로는 같은 GenTL Producer(.cti)를 거쳐 카메라와 통신하며, 차이는 "그 위에 어떤 사용자 경험과 개발 자유도를 제공하는가"에 있습니다.

2. 소프트웨어 종류

① 카메라 뷰어 (Camera Viewer)

  • 카메라와 네트워크 환경을 간단하게 설정하고 이미지를 실시간으로 확인할 수 있는 기본 소프트웨어
  • 주요 기능
    • 네트워크 구성 : 카메라를 네트워크에 연결하고 IP 설정
    카메라 구성 : 노출, 게인, 해상도 등 카메라 파라미터 설정
    이미지 캡처 / 향상 / 표시 / 기록
    분석 도구 : 히스토그램, 라인 플롯(Line Plot) 등 이미지 분석 기능 제공
개발자 노트 — 코드를 한 줄도 쓰기 전에, 카메라 뷰어로 먼저 노출·게인·트리거가 원하는 대로 동작하는지 확인하는 것이 정석입니다. 뷰어에서 재현 안 되는 문제는 십중팔구 소프트웨어 로직이 아니라 카메라·조명·설치 쪽 문제입니다.

② 포괄적인 소프트웨어 (Comprehensive Software)

  • 카메라 뷰어의 모든 기능을 포함하며, 복잡한 멀티카메라 환경과 고급 이미지 처리를 위한 확장 기능을 제공
  • 주요 기능
    • 프로젝트 관리 : 검사 프로젝트 단위로 설정 저장 및 관리
    다중 카메라 / 서버 구성 : 여러 대의 카메라를 동시에 제어하고 구성
    GPU 지원 : GPU 가속을 활용한 고속 이미지 처리
    3D 이미지 재구성 : 2D 이미지 데이터를 기반으로 3D 형상 생성
    3D 개체 뷰어 : 재구성된 3D 데이터를 시각적으로 탐색

③ SDK (Software Development Kit)

  • 개발자가 자체 애플리케이션에 카메라 기능을 통합할 수 있도록 제공되는 개발 도구 모음
  • 주요 구성 요소
    • 코드 샘플 : 주요 기능 구현을 위한 예제 코드 제공
    라이브러리 : 카메라 제어 및 이미지 처리를 위한 함수 라이브러리
    API : 카메라와 소프트웨어 간 인터페이스 명세
    문서 : 상세 사용 설명서 및 레퍼런스 매뉴얼
    컴파일러 / 디버거 : 개발 및 디버깅 환경 지원
  • 지원 언어 : C / C++ / C# / Python
  • 지원 플랫폼 : Windows / Linux / ARM
개발자 노트 — 제조사 전용 SDK는 성능·기능이 풍부하지만 특정 카메라에 종속됩니다. 여러 제조사 카메라를 함께 다루거나 장기적으로 카메라 교체 가능성이 있다면, 3절에서 다루는 GenICam/GenTL 표준 기반 오픈소스 라이브러리(예: Harvester) 위에 애플리케이션을 구성하는 것도 좋은 대안입니다.

3. 표준 이미지 획득 파이프라인

  • 제조사·인터페이스(GigE/USB3/CXP)가 달라도, GenICam 기반 애플리케이션의 이미지 획득 흐름은 대부분 아래와 같은 공통 단계를 따름
개발자 노트 — ⑤~⑧ 구간(버퍼 획득 → 처리 → 반환)이 카메라의 프레임 레이트 안에 끝나지 않으면 내부 버퍼가 가득 차 프레임 드랍이 발생합니다. 검사 알고리즘이 무거워질 것에 대비해, 획득(⑤)과 처리(⑥~⑦)를 서로 다른 스레드로 분리하는 설계를 기본으로 가져가는 것이 안전합니다 (4절 참고).

4. 버퍼 관리와 멀티스레드 아키텍처

  • 폴링(Polling) 방식 : 애플리케이션이 주기적으로 "새 이미지가 있는지" 확인하며 가져오는 방식. 구현이 단순하지만 폴링 주기에 따라 지연이 생길 수 있음
  • 콜백(Callback/Event) 방식 : 새 이미지가 도착하면 드라이버가 등록된 콜백 함수를 즉시 호출. 지연이 적어 고속 스트리밍에 유리하지만 콜백 함수 내부 처리 시간 관리가 중요
  • 일반적인 실무 아키텍처 : 획득 스레드(카메라에서 버퍼를 최대한 빨리 fetch해 큐에 적재)와 처리 스레드(큐에서 이미지를 꺼내 검사 알고리즘 수행)를 분리
    • 획득 스레드가 처리 지연에 영향받지 않도록 하여 프레임 드랍을 최소화
    처리 속도가 획득 속도를 따라가지 못하면 큐에 이미지가 쌓이므로, 큐 최대 길이를 정해 오래된 프레임을 버리는 정책(Drop Oldest)도 함께 설계
개발자 노트 — 카메라 연결이 순간적으로 끊기거나 재연결되는 상황(케이블 접촉 불량, 스위치 재부팅 등)에 대한 예외 처리와 자동 재연결 로직은 초기 설계에서 빠뜨리기 쉽지만, 실제 생산 라인에서 24시간 가동할 때 안정성을 좌우하는 핵심 요소입니다.

5. 이미지 처리 라이브러리 생태계

라이브러리특징주 용도
OpenCV오픈소스 무료, 방대한 커뮤니티, 범용 컴퓨터비전 알고리즘커스텀 알고리즘 개발, 예산 제약이 있는 프로젝트
MVTec HALCON2500개 이상의 연산자, GPU 가속, 3D 비전, 다양한 임베디드 플랫폼 지원고성능 산업 검사, 로봇 가이던스
Cognex VisionPro드래그앤드롭 GUI, 룰 기반+딥러닝 툴 결합빠른 애플리케이션 구성, EtherNet/IP 연동
Euresys Open eVision패턴 매칭·정렬·측정 등 모듈형 라이브러리프레임그래버 연동 검사 시스템
개발자 노트 — 딥러닝 기반 검사가 필요하면 컴퓨팅 페이지에서 다룬 TensorRT/ONNX Runtime 같은 추론 엔진과 위 라이브러리를 조합하는 경우가 많습니다(예: HALCON의 내장 딥러닝 툴, 또는 OpenCV로 전처리 후 별도 추론 엔진 호출). 상용 라이브러리는 런타임 라이선스 비용이 카메라 대수만큼 발생하는 경우가 많으니, 초기 견적 단계에서부터 라이선스 정책을 확인해야 합니다.

6. 머신비전 개발자를 위한 소프트웨어 스택 체크리스트

  • 표준 준수 — GenICam/GenTL 호환 여부로 카메라 교체·멀티벤더 대응 유연성 확보
  • 획득 아키텍처 — 획득 스레드와 처리 스레드 분리, 버퍼 큐 정책(오래된 프레임 처리 방식) 설계
  • 예외 처리 — 카메라 연결 끊김, 타임아웃, 재연결 시나리오에 대한 복구 로직
  • 이미지 처리 라이브러리 — 예산·성능·GUI 필요 여부에 따라 OpenCV vs 상용 라이브러리(HALCON/VisionPro 등) 선택
  • 딥러닝 통합 — 추론 엔진(TensorRT/ONNX Runtime)과 기존 파이프라인의 연동 방식
  • 라이선스 비용 — 상용 SDK/라이브러리의 카메라 대수·PC 대수 기준 라이선스 정책 사전 확인
